# Contributing

## 목적

이 문서는 `dx12_Graphics` 저장소에 기여할 때 따라야 할 기본 작업 흐름을 정리합니다.
현재 프로젝트는 초기 구축 단계이므로, 빠른 기능 추가보다 일관된 구조와 안정적인 변경 절차를 우선합니다.

## 기본 원칙

- 변경은 가능한 한 작은 단위로 나눕니다.
- 항상 빌드 가능한 상태를 유지합니다.
- 구현과 동시에 테스트 또는 검증 방법을 함께 준비합니다.
- 구조 변경 시에는 코드뿐 아니라 관련 문서도 함께 갱신합니다.

## 작업 흐름

1. 작업 목적과 범위를 정리합니다.
2. 필요한 경우 관련 문서를 먼저 확인합니다.
3. 작은 단위로 변경을 진행합니다.
4. 변경 내용을 스스로 점검한 뒤 리뷰를 요청합니다.
5. AI 리뷰와 사람 리뷰 의견을 반영합니다.

## 코드 작성 시 기대 사항

- 새 코드는 기존 구조와 네이밍 규칙을 따릅니다.
- DirectX 12 리소스 수명과 상태 전이를 명확하게 드러냅니다.
- 테스트가 어려운 변경이라면 최소한 수동 검증 절차를 남깁니다.
- 임시 코드나 디버그 코드는 목적이 끝나면 정리합니다.

## 리뷰 요청 시 기대 사항

리뷰 요청 시에는 가능한 한 아래 내용을 함께 공유합니다.

- 변경 목적
- 주요 변경 포인트
- 확인한 빌드 구성
- 테스트 또는 수동 확인 방법
- 남아 있는 제한 사항 또는 후속 작업

## 커밋과 PR 방향

- 한 커밋에는 하나의 논리적 변경을 담는 것을 권장합니다.
- PR은 가능한 한 하나의 주제를 중심으로 구성합니다.
- 스타일 정리와 기능 변경이 섞이는 경우에는 분리하는 것을 우선 고려합니다.
- 커밋 메시지는 `docs/commit-message.md`에 정의된 규칙을 따릅니다.

## AI Review And Slack

- 향후 이 저장소는 AI 기반 코드 리뷰와 Slack 알림 흐름을 함께 사용할 예정입니다.
- 따라서 PR 설명, 변경 범위, 검증 절차를 명확하게 남기는 것을 기본 원칙으로 합니다.
