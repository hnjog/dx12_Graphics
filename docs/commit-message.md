# Commit Message Rules

## 목적

이 문서는 `dx12_Graphics` 저장소에서 사용하는 커밋 메시지 규칙을 정의합니다.
목표는 커밋 이력을 빠르게 읽을 수 있게 하고, 변경 의도와 범위를 한눈에 파악할 수 있도록 하는 것입니다.

## 기본 형식

커밋 메시지는 아래 형식을 기본으로 사용합니다.

`[type] 수정 파일 또는 작업 대상 - 세부 내용`

예시:

- `[init] 프로젝트 문서 - README 및 기본 가이드 추가`
- `[add] dx12Engine - swap chain 초기화 코드 추가`
- `[fix] renderer - fence 동기화 순서 수정`

## 타입 정의

- `init`: 초기화, 최초 생성, 기본 구조 세팅
- `add`: 새로운 파일, 기능, 모듈 추가
- `del`: 파일 제거, 기능 제거, 불필요한 코드 삭제
- `update`: 기존 기능 확장, 동작 개선, 문서 내용 보강
- `fix`: 버그 수정, 안정화, 리팩터링, 잘못된 동작 수정

## 작성 원칙

- 한 커밋에는 가능한 한 하나의 주제만 담습니다.
- 어떤 파일을 바꿨는지보다 왜 바꿨는지가 보이도록 작성합니다.
- 너무 긴 문장보다는 핵심만 간결하게 작성합니다.
- 여러 파일을 수정했더라도 하나의 목적이라면 하나의 대상 표현으로 묶습니다.
- 단순 포맷 정리와 기능 변경은 가능한 한 분리합니다.

## 추천 패턴

- 문서 작업: `[update] docs - 리뷰 규칙 문서 보강`
- 초기 구성: `[init] project - 기본 문서 및 ignore 규칙 추가`
- 기능 추가: `[add] renderer - command queue 생성 로직 추가`
- 구조 개선: `[update] engine - 리소스 관리 구조 정리`
- 버그 수정: `[fix] dx12Engine - null pointer 검사 순서 수정`
- 제거 작업: `[del] sample - 사용하지 않는 테스트 코드 제거`

## 현재 작업에 어울리는 예시

현재까지의 변경 사항을 기준으로 하면 아래와 같은 커밋 메시지가 자연스럽습니다.

- `[init] project docs - README 및 기본 운영 문서 추가`
- `[update] docs - 코딩 규칙과 리뷰 규칙 문서 정리`
- `[init] repository - gitignore 및 프로젝트 기본 문서 구성`

## 비고

- 현재는 `init/add/del/update/fix` 다섯 가지 타입을 기본으로 사용합니다.
- 팀 합의가 생기면 타입을 더 늘리거나 세분화할 수 있습니다.
- 표기 혼선을 줄이기 위해 타입은 소문자로 통일하는 것을 권장합니다.
