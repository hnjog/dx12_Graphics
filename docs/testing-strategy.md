# Testing Strategy

## 목적

이 문서는 `dx12Engine` 프로젝트에서 어떤 방식으로 빌드, 실행, 수동 검증, 회귀 확인을 진행할지에 대한 기본 전략을 정리합니다.
현재는 자동화보다 재현 가능하고 기록 가능한 검증 절차를 먼저 만드는 것을 목표로 합니다.

## 기본 원칙

- 변경 후에는 최소한 빌드 가능 여부를 확인합니다.
- 렌더링 결과가 중요한 기능은 수동 확인 절차를 함께 남깁니다.
- 버그 수정은 가능한 한 재현 조건과 수정 후 확인 절차를 같이 기록합니다.
- 테스트가 어려운 단계라도 무엇을 확인했는지는 PR 또는 작업 기록에 남깁니다.

## 우선 대상 구성

현재 단계에서는 아래 구성을 우선 확인 대상으로 삼습니다.

- `Debug | x64`
- `Release | x64`

`Win32` 구성은 프로젝트에 존재하더라도, DirectX 12 엔진 개발의 주력 검증 대상은 우선 `x64`로 둡니다.

## 테스트 수준

### Build Check

- 프로젝트가 정상적으로 빌드되는지 확인합니다.
- 필수 헤더, 라이브러리, 프로젝트 설정 변경이 의도대로 반영되었는지 확인합니다.

### Startup Check

- 프로그램이 실행 직후 종료되지 않는지 확인합니다.
- 윈도우 생성, Device 초기화, Swap Chain 초기화 같은 시작 단계가 정상 동작하는지 확인합니다.

### Rendering Check

- 초기 clear color 출력
- 프레임 갱신 여부
- 창 크기 변경 시 기본 동작
- command submission 및 present 흐름의 기본 동작

### Regression Check

- 기존에 동작하던 초기화 흐름이 깨지지 않았는지 확인합니다.
- fence, resource state, descriptor 관련 변경은 기존 렌더링 경로에 영향이 없는지 확인합니다.

## 수동 테스트 기록 예시

변경 후에는 가능한 한 아래 형태로 확인 내용을 남깁니다.

- 확인 구성: `Debug | x64`
- 실행 결과: 정상 실행 / 실행 실패
- 확인 항목: 윈도우 생성, 초기 clear, 종료 동작
- 특이 사항: warning, validation message, known issue

## 버그 수정 시 기대 사항

- 어떤 상황에서 문제가 발생했는지 기록합니다.
- 수정 후 어떤 방식으로 다시 확인했는지 남깁니다.
- 재현 가능한 경우 재현 절차를 짧게라도 문서화합니다.

## 향후 확장 방향

- 테스트 프로젝트 분리
- 샘플 실행 기반 회귀 체크
- CI 빌드 자동화
- AI 리뷰 결과와 테스트 결과의 연결
- Slack 알림과 빌드 상태 공유
