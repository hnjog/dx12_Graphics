# Coding Standard

## 목적

이 문서는 `dx12Engine` 프로젝트의 C++ 및 DirectX 12 코드 작성 규칙을 정의합니다.
목표는 단순한 스타일 통일보다는 안정성, 디버깅 용이성, 유지보수성, 그리고 렌더링 파이프라인의 명확성을 확보하는 것입니다.

## 문서 작성 원칙

- 문서 설명은 기본적으로 한국어를 사용합니다.
- 코드 식별자, API 이름, 도구 이름, 파일명은 영어 원문을 유지합니다.
- 한국어보다 영어 표현이 더 정확한 경우에는 `RAII`, `Command Queue`, `Resource State`처럼 영어 용어를 그대로 사용합니다.

## 적용 도구

- `.editorconfig`는 줄 끝, 들여쓰기, 공통 편집 규칙을 관리합니다.
- `.clang-format`은 코드 포맷 규칙을 관리합니다.
- `.clang-tidy`는 정적 분석과 일부 코드 품질 규칙 점검에 사용합니다.
- 모든 규칙이 도구만으로 완전히 강제되지는 않으므로, 리뷰 규칙과 함께 운영합니다.

## 일반 원칙

- 엔진 코드와 테스트 코드는 역할을 분리합니다.
- 구현 속도보다 리소스 수명과 상태 전이의 명확성을 우선합니다.
- 암묵적 동작보다 명시적 동작을 선호합니다.
- 하나의 파일과 하나의 타입은 가능한 한 하나의 책임에 집중합니다.
- 자동화 가능한 규칙은 문서에만 두지 않고 향후 도구 설정으로도 관리합니다.

## 네이밍

- 타입, 클래스, 구조체, enum: `PascalCase`
- 함수: `PascalCase`
- 지역 변수와 매개변수: `camelCase`
- 멤버 변수: `m_camelCase`
- 상수: 팀 규칙이 확정되기 전까지 `PascalCase` 또는 `kPascalCase` 중 하나로 통일해서 사용합니다.
- 매크로: 반드시 필요한 경우에만 사용하며 `UPPER_SNAKE_CASE`를 사용합니다.

## Condition And Pointer Checks

- Boolean 조건은 암묵 평가보다 명시 비교를 사용합니다.
- `if (isVisible == true)` 또는 `if (isFinished == false)` 형태를 사용합니다.
- 포인터 조건은 암묵 평가보다 `nullptr`와의 명시 비교를 사용합니다.
- `if (device == nullptr)` 또는 `if (commandQueue != nullptr)` 형태를 사용합니다.
- 이 규칙은 일부 정적 분석으로 보조할 수 있지만, 최종적으로는 코드 리뷰에서도 함께 확인합니다.

## C++ Rules

- `raw new/delete` 사용을 피하고 RAII를 우선합니다.
- 소유권이 있는 리소스는 스마트 포인터 또는 전용 래퍼 타입으로 관리합니다.
- 함수가 실패할 수 있다면 실패 경로를 숨기지 말고 명확하게 드러냅니다.
- 헤더에는 필요한 선언만 두고, 구현 세부사항은 소스 파일에 둡니다.
- `using namespace`는 전역 범위에서 사용하지 않습니다.
- 설명이 필요한 매직 넘버는 상수로 치환합니다.

## DirectX 12 Rules

- COM 객체는 일관된 래퍼 또는 `ComPtr` 계열로 관리합니다.
- 리소스 상태 전이는 호출 지점에서 명시적으로 보이도록 작성합니다.
- Descriptor Heap, Command List, Command Queue의 소유 주체를 분명히 구분합니다.
- 프레임 단위 리소스와 장기 수명 리소스를 혼합하지 않습니다.
- Debug 빌드에서는 가능한 한 DX12 Debug Layer와 검증 메시지를 활용합니다.

## 프로젝트 구조

- 엔진 코어와 렌더링 백엔드를 분리합니다.
- 플랫폼 의존 코드와 렌더링 로직을 직접 섞지 않습니다.
- 테스트 또는 샘플 코드는 엔진 API 사용 예시가 되도록 유지합니다.

## Logging And Assertions

- 복구 불가능한 상태는 조용히 무시하지 않고 assert 또는 명시적 실패로 처리합니다.
- 로그는 원인 추적에 도움이 되는 정보 중심으로 남깁니다.
- 프레임마다 과도하게 반복되는 로그는 제한합니다.

## Testing

- 새 기능에는 가능한 한 최소 하나 이상의 검증 경로를 함께 둡니다.
- 렌더링 결과 확인이 필요한 경우 실행 절차를 문서화합니다.
- 버그 수정 시 재현 조건 또는 확인 절차를 PR 설명에 남깁니다.
